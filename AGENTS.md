## 项目介绍

本项目是一个 **AI for Protein Design（人工智能驱动蛋白质设计）** 的实验平台。基于 RFdiffusion → ProteinMPNN → ESMFold/Boltz-2 → ESM 管线，覆盖从蛋白质骨架生成、序列设计到结构验证的完整闭环。

项目分为三大部分：

1. **环境搭建** — 通过 opencode + SSH 将计算工具自动部署到 AutoDL 云主机（GPU），使任何人在任何新机器上都能一键复现完整实验环境。

2. **实验矩阵** — `Experiments/` 下按设计任务组织为 10 个集团、65 个实验目录，覆盖 Binder 设计、酶设计、抗体设计、环肽设计、金属结合蛋白、对称寡聚体、部分扩散优化、序列属性工程以及前沿课题。

3. **知识库** — 所有 Markdown 文件（`DESIGN_GOALS.md`、`MANUAL.md`、`AMINO_ACID_CODE.md` 等）记录设计理论、操作手册和参考资料。

## 环境配置

当用户提出**配置实验环境**（如"帮我搭建环境""配置云主机""安装工具"等）时，读取并执行 `AgentGuides/ENVIRONMENT_SETUP.md`，注意用户很有可能并不是从零开始，此时你要通过**验证各个工具的存在性**来判断当前哪些步骤已被执行，哪些步骤将被执行。如果用户提出安装**新工具**（不存在于上述文件），将其视作一次变更。此时每新做一步都需要**实时记录**和**变更日志**，并且**绝对不允许在合并之前直接修改AGENTS.md或者AgentGuides/ENVIRONMENT_SETUP.md**。

## 云端环境

本项目的计算在 AutoDL 云主机上执行。你的所有云端操作务必遵循以下磁盘布局：

| 路径                  | 别名   | IO速度 | 用途       | 保存镜像时 |
| ------------------- | ---- | ---- | -------- | ----- |
| `/`                 | 系统盘  | 一般   | conda 环境 | 会保留   |
| `/root/autodl-tmp/` | 数据盘  | 快    | 代码、权重、输出 | 不会保留  |
| `/root/autodl-fs/`  | 文件存储 | 一般   | 跨实例共享    | 不受影响  |

核心原则：**系统盘仅 30 GB**。所有大文件——代码仓库、模型权重、实验输出——一律放 `/root/autodl-tmp/`（50 GB）。conda 环境若体积过大，也挪到该盘并用 symlink 挂回系统盘。如果遇到无 GPU 启动，这是正常的省钱模式，代码验证仍可继续，但实际推理需用户切换到有 GPU 模式。

关机时所有文件保留，但再次开机时可能遇到无可用GPU的问题。此时可以告诉用户有一种选择：克隆实例。可以将实例克隆到有GPU的新主机。可能需要一些时间。克隆完成之后两实例完全相同，原有的无GPU的实例可以删除。但注意这种操作限制一天最多10次。并且会遇到`AgentGuides/Situations/26.05.29:DUPLICATE_BUG.md` 里面提到的问题。

## 实验注意

每次实验所产生的新文件，永远要在实验对应的文件夹内，不可以散落在外面。这个原则应用于每次实验的EXPERIMENT.md的制定。比如，Exp4.3:TetrahedralOligomer/这个实验的ProteinMPNN产生的结果就应该在云端的Exp4.3:TetrahedralOligomer/outputs/proteinmpnn里面。

## 杂项

### 命名逻辑

- 所有给**具有自然语言理解能力**的个体——包括人类和AI Agents——阅读的**Markdown文件**全部使用**全大写**，**下划线分隔**的命名方式，比如DESIGN_GOALS.md。

- 所有能被**编译器或解释器**通过严格的**语法解析**的各种**文件**全部使用**全小写**，**下划线分隔**的命名方式，比如download_weights.py。

- 所有的**目录名称**使用**仅首字母大写**，**单词直接相连**的命名方式，比如AgentGuides/。

- 特别地，当一组文件/目录需要某种**序号**来标注以便被检索和排列的时候，使用<序列号>:<符合前一条规则的文件/目录名字>，比如EG2:MotifScaffolding/和Exp4.3:TetrahedralOligomer/。

### 实时记录

当你在某种条件限制下，出现了**本文件AGENTS.md或AgentGuides/** 中没有写过的异常情况，或者被迫执行与**本文件AGENTS.md或AgentGuides/** 中所述不相同的操作，请一定要记录！！以便复现或将其纳入到我们的标准操作中。记录在**AgentGuides/Situations/** 目录下，文件命名格式为xx.xx.xx:XXXX.md。这里冒号之前的三个xx分别代表年、月、日，冒号之后是你对这一特殊情况的命名。注意，由于一天可能执行很多操作，同一个异常情况（即你打算叫同样名字的情况）可能会出现多次导致你想多次创建文件。所以请你记住：**永远在不存在时才创建**，**永远追加**，**每次追加永远最后用---结尾**。记录后**告知用户**，并在合适时机提醒用户审阅。

### 变更日志

与 Situations 的事件驱动不同，变更日志是纯流水账，记录**每一次变更操作**（无论成败），用于追溯历史。记入 `AgentGuides/Logs/CHANGE_LOG.md`，格式：

```
YY.MM.DD HH:MM | <操作> | <结果> | <备注>
---
```

**只能追加**。

### 安全变更

当用户要求对**已有可工作**的代码/配置做改动时，禁止原地修改。按以下步骤操作：

1. **新建验证版本** — 若原文件为 `script.py`，创建 `script_v2.py` 写入新方案。原文件不动，本身就是回滚目标
2. **独立验证** — 先在新文件上单独跑，确认自身逻辑正确
3. **集成验证** — 确认新文件与管线中上下游模块协同工作无误（例如：换了 ProteinMPNN 版本后，验证 RFdiffusion → 新版 MPNN → ESMFold 全链路通过）
4. **替换或回滚** —
   - 两步验证均通过 → `mv script_v2.py script.py`，并记录至**变更日志**。
   - 任一验证失败 → 删 `script_v2.py`，保留原文件，将失败原因记录到 `AgentGuides/Situations/`
