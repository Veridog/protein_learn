# fold_all.py seq_idx 计算错误导致 37.5% 序列漏折

## 现象

Exp0.1 中 ESMFold 只折叠了 500/800 条序列（期望 800）。缺失的 300 条恰好是每个 backbone 的 seq_idx 5-7（全部为 0）。

## 根因

`fold_all.py` 中 `parse_mpnn_fasta()` 返回 `(header, seq)` 元组列表（已配对好），迭代时：

```python
for si, (header, seq) in enumerate(seqs):
    if si == 0:
        continue
    seq_idx = si // 2      # ← BUG
```

`seqs` 有 9 个元素（1 backbone + 8 designed）。跳过 backbone (si==0) 后，`si // 2` 导致：

| si | si // 2 | 正确值 (si-1) |
|----|---------|---------------|
| 1  | 0       | 0             |
| 2  | 1       | 1             |
| 3  | 1       | **2**         |
| 4  | 2       | **3**         |
| 5  | 2       | **4**         |
| 6  | 3       | **5**         |
| 7  | 3       | **6**         |
| 8  | 4       | **7**         |

seq_idx 1-4 各被分配到 2 条不同的序列。第二条序列因检测到 `design_X_seqs_seqY.pdb` 已存在被 `continue` 跳过。seq_idx 5-7 从未有序列被分配到。

### 影响范围

- Exp0.1 损失 300 条预测结构
- 后续 Exp0.2 / Exp0.3 使用的数据不完整（但也够用，500 条仍可观）

## 修复

```python
# 修复前
seq_idx = si // 2

# 修复后
seq_idx = si - 1  # skip backbone at si=0
```

修复后需删除已有的 500 个 PDB 重新全量折叠，或仅补充折叠 seq_idx 5-7。

### 为什么会写出 // 2？

这是典型的模式误用。`// 2` 常见于按行解析（header 行 + seq 行交错），但这里的 `seqs` 已经是成对组装的列表了。
